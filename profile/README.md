## 评测维度说明

要客观对比"线路加速"和"软件加速"这两条路，我们需要先定义几个判断标准：

**延迟（Ping 值）**：最直观的体感指标，影响页面响应和实时交互。软件加速几乎无法改善延迟，这是物理距离决定的。

**晚高峰稳定性**：晚上 8 点到 11 点，国际出口最拥堵。CN2 GIA 等高端线路有优先级保障，普通线路在这个时段掉速明显。

**丢包率**：网络质量的核心指标。BBR 等算法能在一定丢包环境下尽量跑满带宽，但本身不减少丢包。

**带宽上限**：套餐硬件决定，软件无法突破。

**价格门槛**：综合考虑长期使用成本。

## 路径一：软件加速（BBR / BBRplus / 锐速）

### 它能做什么

BBR 是 Google 开源的 TCP 拥塞控制算法，2016 年提出，原理是在丢包环境下更积极地占用带宽，同时降低缓冲延迟。对搬瓦工用户来说，好消息是现在新装的 Linux 系统基本默认已经开启了 BBR，不需要额外操作。

如果你想进一步折腾，还有 BBRplus、BBR 魔改版、Lotserver 锐速等选项。一键安装脚本很成熟，SSH 连上去，跑几条命令，几分钟就完事。安装完记得重启，不重启不会生效。

锐速（ServerSpeeder/Lotserver）和 BBR 有点互补：BBR 更擅长提升下载速度，锐速在上传场景表现稍好。可以都试试，看哪个对你的网络环境更友好。

### 它解决不了什么

说白了，BBR 是在"次优线路"里尽量多抢带宽，它没法改变路由路径，没法降低延迟，也没法解决晚高峰拥塞——因为拥塞本身是出口节点的问题，不是你这台 VPS 的 TCP 算法问题。

如果你的套餐走的是普通 163 骨干网，装再好的加速脚本，晚上 10 点该卡还是卡。

### 适用场景

软件加速更适合这种情况：线路本身质量还行，就是想在现有基础上再压榨一点性能。或者套餐线路对丢包不太敏感的场景，比如下载大文件、批量数据传输等。

## 软件加速怎么装（给已有合适套餐的用户）

如果你已经在用 CN2 GIA-E 或更高线路，想在此基础上进一步优化，可以试试以下步骤：

1. SSH 连接到你的搬瓦工 VPS（工具用 Termius、FinalShell 或系统终端均可）
2. 检查当前系统是否已开启 BBR：运行 `sysctl net.ipv4.tcp_congestion_control`，输出包含 `bbr` 说明已开启
3. 如需安装 BBRplus 或锐速，使用多合一加速脚本（GitHub 上可找到），运行后按菜单数字选择对应方案
4. 安装完成后**务必重启**，不重启不生效
5. 重启后重新检查拥塞算法，再做一次测速对比

BBR 和锐速没有绝对哪个更好，取决于你的网络环境和使用场景——最稳妥的方法是都试一遍，留下更合适的那个。
